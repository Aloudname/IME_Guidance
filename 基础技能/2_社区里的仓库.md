# 2 社区里的仓库

@author(Aloudname)

这一节我们学如何跟别人的仓库打交道，下载仓库、提交反馈、创建分支、提交合并申请等。

## 1. 使用 clone 下载别人的仓库

将别人仓库里的代码下载到本地（自己的开发环境），这就是所谓的 `clone`。GitHub 提供了方便的几种 clone 办法。

![repoClone](../src/基础技能/2_社区里的仓库/methods_clone.png)

第一种是直接下载打包的.zip文件，然后手动解压；

第二种是直接在 VSCode 编辑器中打开；

第三种我最常用，复制其 SSH 命令，然后在目标位置运行命令；

第四种方法使用我们在 [GitHub使用.md](1_GitHub使用.md) 里提到的 GitHub CLI:

```bash
gh repo clone <用户名>/<仓库名>
```

拿第三种（SSH）举例：

```bash
# 进入要下载到的目标路径里
cd /d <目标路径>

# 命令格式：git clone <git地址>
git clone git@github.com:rmcong/RGBD-Cosal150-Dataset.git
```

这种方法需要本地安装 [GitHub使用.md](1_GitHub使用.md) 里提到的 `git` 服务并且配置好 GitHub 账户。只需配置一次，后面非常方便。

注意，这些方法只是下载了别人的代码。如果你想基于别人的仓库创建一个属于自己的拷贝仓库，需要用到下面的 `fork`。

## 2. 使用 fork 创建拷贝

我们没有直接修改别人仓库的权限。如果我们想贡献内容，怎么办？首先要创建属于自己的分支（`fork`）。这个分支其实就是源仓库的一个拷贝，只不过所有权属于自己。我们可以对这个分支随意做更改，与源仓库互不影响（注意，源仓库的改动也不会影响分支）。借助这个特性，我们可以用分支学习、扩展别人的源代码，这就是 `fork` 的意义。

在我们想拷贝的源仓库，点击 `Fork` 按钮，并在新页面选择将分支仓库的所有权划给谁（我们的个人账户，或者我们加入的某组织），最后点击 `Create fork`。这样我们就创建了一个可以直接改动的分支仓库。

![try fork](../src/基础技能/2_社区里的仓库/fork_1.png)
![choose owner](../src/基础技能/2_社区里的仓库/fork_2.png)

## 3. 使用 pull requests 贡献内容

我觉得自己新增的内容很不错，比原作者的水平高十分甚至九分。这么好的程序，若不发出去岂不可惜！这时候我该怎么做呢？确认好自己的更改已经上传到 GitHub 上的分支仓库后，我们进入分支仓库。这时候会出现一个状态栏，提示

> “This branch is xxx commit ahead of <源仓库>”

也就是说我们的分支相比源仓库有了新的改动。我们可以点击这个状态栏里的 `contribute` ，也可以点击上方的 `pull requests`。这两个按钮最终都会将我们定向到新建一个合并请求的界面。

![our fork](../src/基础技能/2_社区里的仓库/pull_request_1.png)

比如我们点击了 `pull requests`，就会进入下面的界面。点击 `New pull request` 创建一个新的合并请求：

![pull requests](../src/基础技能/2_社区里的仓库/pull_request_2.png)

这样，系统就会给我们列出这个分支相对于源仓库的改动。另外，红框框中还可以选择将分支仓库的哪个 `branch` 合并到源仓库的哪个 `branch` 里。对初学者来说，一般情况下都是唯一的主 branch（叫 `main`） 合并到主 branch `main` 中，不必改动。多 branch 经常在多版本开发（比如不同游戏版本 / 不同衍生功能的程序）和大型开发（一个正式发布 branch，一个测试开发 branch 等）中使用。检查 branch 和更改的内容没问题后（请务必做这件事），点击 `Create pull request`：

![current requests](../src/基础技能/2_社区里的仓库/pull_request_3.png)

这个页面需要填写这次合并的信息，是展示给源仓库所有者用的，因此可以填写得详细一点。如果懒，可以点击红色框框，让 Copilot 帮你写一份。主要说明你做改动的地方以及原因。如果有新函数/变量/类等一切新内容，请不吝键盘给出其类型、用法和结构。这个页面和 `issues` 也是一样的，支持 `.md` 文本。

![create it](../src/基础技能/2_社区里的仓库/pull_request_4.png)

以下是一个对 PyTorch 库发起合并请求的实例。请求的发起者完善了 PyTorch 的注释内容（微不足道但也重要），在合并请求中说明了改动的性质（docs，对应文档修改）和改动的内容（解释某调试文件）。

![try torch](../src/基础技能/2_社区里的仓库/pull_request_5.png)

## 4. 提交 issues

issues 是非常方便的反馈方法。直接在源仓库点击 `issues` 进入，这里可以看见其他人提交的反馈，还有仓库所有者的回复。点击 `new issue`，填写标题和正文，最后点击 `create` 就能提交啦！正文其实就是一个 `.md` 文件，因此支持各种 markdown 组件和功能。红色的框框可以让 GitHub 内置的 Agent —— Copilot 帮忙写。
